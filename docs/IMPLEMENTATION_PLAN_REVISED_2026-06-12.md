## 全体方針（2026-06 時点）

1. **撮影体験の速さは維持** — フィードバックで評価が高い核は変えない。
2. **失敗時はユーザー主導** — 自動リトライで API 消費・待ち時間を増やさず、明示的な「再送信」に寄せる。
3. **修正は手入力追加と同じ UI** — 要確認・通常修正の境界を薄くし、画像参照しながら品目追加できるようにする。
4. **無駄遣い度は AI 基準の強化** — 手動5段階評価は採用せず、プロンプトと例示で判定を揃える。
5. **ナビゲーションの期待動作を先に直す** — 日常利用で毎回触る導線の不具合を最優先。

## Phase 6: 実利用フィードバック対応（新規）

> 目標: **1か月利用で顕在化した UX ギャップと不具合を解消する**

### 6.1 ナビゲーション修正（最優先・小〜中）

**背景**: `NavHost` の `restoreState` / 詳細画面がバックスタックに残るため、ランチャー再起動やタブ切替の期待とずれる。

- [ ] **6.1.1 ランチャーアイコン → 常にカメラタブ**
  - `MAIN/LAUNCHER` で起動したとき（コールドスタート・既存タスクの前面復帰の両方）カメラを表示
  - 詳細・修正・プレビュー等の子画面は pop する
  - 実装案: `MainActivity.onCreate` / `onNewIntent` で「ランチャー起動」フラグを判定し、`navController` をカメラへ `popUpTo` + `launchSingleTop`
  - 通知タップ（`EXTRA_RECEIPT_ID` あり）のときだけ詳細へ遷移（現状維持）

- [ ] **6.1.2 ボトムタブ切替で子画面を pop**
  - 詳細・修正・プレビュー上にいる状態で **任意のタブ**（特にカメラ）を押したら、該当タブのルートへ戻る
  - 実装案: タブ `onClick` で `popUpTo(navController.graph.startDestinationId) { saveState = true }` の後に目的タブへ `navigate`
  - 一覧タブの月フィルタ等 `rememberSaveable` 状態は維持

**完了条件**: ランチャー再起動 → カメラ表示。通知 → 詳細 → カメラタブ → カメラ表示。

**関連コード**: `MainActivity.kt`（`AppNav` のタブ `onClick`、`LaunchedEffect(deepLinkReceiptId)`）

---

### 6.2 解析失敗の手動再送信（高優先・中）

**背景**: `AnalysisWorker` は現状、失敗時に WorkManager 自動リトライ（合計2トライ）。ユーザーは API 制限下で **能動的リトライ** を希望。

- [ ] **6.2.1 自動リトライの停止**
  - 1回目失敗で `Result.retry()` せず `FAILED` 確定
  - `analysisErrorMessage` にユーザー向け文言を保存

- [ ] **6.2.2 再送信 UI**
  - 表示箇所: レシート詳細（`analysisStatus == FAILED`）、通知タブ（失敗履歴/キュー）、必要なら一覧行の状態表示
  - 操作: 「再送信」ボタン → 確認（任意）→ `analysis_queue` に再投入（`QUEUED`、`attemptCount` リセット）
  - 再投入後は既存 Worker フローで1回だけ解析

- [ ] **6.2.3 再送信のガード**
  - API キー未設定時は設定へ誘導
  - 画像ファイル欠落時はエラー表示（再撮影/削除を案内）
  - 連打防止（ボタン disabled / クールダウン）

**完了条件**: 失敗レシートに「再送信」が表示され、タップで解析が再開される。自動リトライは行われない。

**関連コード**: `AnalysisWorker.kt`、`ReceiptRepository.kt`、`ReceiptDetailScreen.kt`、`NotificationsScreen.kt`

---

### 6.3 修正画面の本格化（高優先・大）

**背景**: `ReceiptReviewScreen` は要確認向け最小編集（必須項目・支払手段・必須度・カテゴリ欠落補完）。ユーザーは **手入力追加（`ManualExpenseScreen`）と同等** の編集を求めている。

- [ ] **6.3.1 レシート画像の常時表示**
  - 修正画面に画像プレビュー（折りたたみ or 上部固定）
  - 詳細画面の `showImageSheet` と同ソース（`receipt_images`）を利用

- [ ] **6.3.2 明細行のフル編集**
  - 商品名・数量・金額・大/小カテゴリ・必須度スライダー（`ManualExpenseScreen` の行エディタを共通化）
  - 既存行の **誤分類修正** も許可（要件 §11 の「初期は絞る」を実利用に基づき緩和）

- [ ] **6.3.3 明細行の追加・削除**
  - 「行を追加」→ 空行を末尾に追加
  - 行削除（確認ダイアログ）
  - 保存時に `lineIndex` 再採番、合計不一致は調整行ロジックを `ReceiptRepository` 既存処理に委譲

- [ ] **6.3.4 詳細画面からの「修正」導線統一**
  - 要確認でなくても修正画面へ（メニュー「修正」）
  - 要確認フローと通常修正で **同一画面** を使う

- [ ] **6.3.5 保存ロジック**
  - `applyReceiptReview` を拡張 or `applyReceiptEdit` を新設
  - 保存後: 必須欠落解消 → `needsReview` 解除、`analysisStatus` を `DONE` に

**完了条件**: 要確認・通常いずれも、画像を見ながら品目追加・編集・保存できる。

**関連コード**: `ReceiptReviewScreen.kt`、`ManualExpenseScreen.kt`（共通化候補）、`ReceiptRepository.kt`

### 6.4 無駄遣い度（necessityScore）基準の明確化（中優先・中）

**背景**: AI 判定とユーザー感覚のズレ。5段階手動評価は入力コスト増のため **不採用**。プロンプトで基準を揃える。

- [ ] **6.4.1 プロンプト改訂（`AnalysisWorker.buildPrompt`）**
  - 境界ケースの **具体例** を追加（例: コンビニ弁当 vs スイーツ、100均雑貨 vs 家具、サブスク通信 vs ゲーム課金）
  - 店種別の暗黙ルールを排除し、「商品の用途」軸を強調（要件 §7 と整合）
  - スキーマ `description` に necessityScore の短い説明を追記（`ReceiptJsonSchema.kt`）

- [ ] **6.4.2 分析タブでの説明補助（任意・小）**
  - 必須/裁量バッジのツールチップ or 設定内ヘルプに「AI による目安。修正画面で調整可」と1行表示
  - 手動5段階入力 UI は **作らない**

- [ ] **6.4.3 回帰用フィクスチャ（最小）**
  - 実利用でズレが大きかったレシート 3〜5 件分の期待スコア帯を JSON で記録（Phase 5.1 継続）

**完了条件**: プロンプト改訂が入り、既知の問題レシートでスコア帯が改善方向に動く（主観確認）。
