# 実装計画（細分化）

この計画は、要件（`docs/REQUIREMENTS.md`）に沿って、実装順を

1. **最低限の動くUI**
2. **Geminiとの連携**
3. **DBと分析**
4. **バックアップ**

の順で進めるためのタスク分解です。

---

## Phase 0: 基盤（横断の前提）

### 0.1 リポジトリ運用
- [ ] ブランチ運用: `feat/*` → PR → `main`（通常マージ）
- [ ] `main`保護: PR必須 + `Android CI` 必須 + force push/delete禁止
- [ ] CIが落ちたときの一次切り分け手順を `docs/` に追記（必要になったら）

### 0.2 アーキテクチャの最小方針
- [x] 画面（Compose）: `ui/*`
- [x] データ（Room/Repository）: `data/*`
- [ ] 外部API（Gemini/Drive）: `integrations/*`（後で追加）
- [ ] 長時間処理（キュー/WorkManager）: `workers/*`（後で追加）

### 0.3 品質/安全性（先にルール化しておく）
- [ ] Roomマイグレーション方針を固定
  - [ ] **開発中は破壊的（destructive）を許容するか**、または **必ずMigrationを書く**かを決める
  - [ ] リリース前には destructive を撤廃し、Migrationで保全する（要件: バックアップ/復元があるため）
- [ ] 端末内ファイルの扱いルール
  - [x] 画像は `filesDir` 管理（外部ストレージに出さない）
  - [x] 一時ファイルは失敗時も掃除（容量リーク防止）
  - [ ] `content://` 由来のURIを扱う場合の方針（削除/読み出し）を決める

## Phase 1: 最低限の動くUI（MVP）

- [x] 画面遷移を下タブ構造に統合（Camera/Preview/Listをタブ配下に置く）

### 1.1 カメラ（CameraX）

- [x] 保存時に **長辺1600pxへリサイズ + JPEG品質85** を適用（要件準拠）

## Phase 2: Geminiとの連携（解析を動かす）

> 目標: **送信確定 → 即キュー投入 → Gemini解析 → DBへ反映**  
> ※ 解析はバックグラウンドで順番に（要件の「1件ずつ」）

### Phase2の前提（決定事項）
- [x] 方針: Phase2で **Receipt上位情報（店名/合計/レシート日時/支払手段）+ 解析状態** までDBに反映する
- [x] 方針: Phase2で **`receipt_items`（明細行）も保存**する（confidence/カテゴリ/必須度をアイテム単位で保持）
- [x] 方針: `analysis_queue.status` は `QUEUED/RUNNING/DONE/FAILED` を採用
- [x] 方針: リトライ回数は **1回（合計2トライ）**
- [ ] 方針: 失敗/低信頼の扱い（暫定）
  - [ ] **厳格JSONとしてパース不能** → `FAILED`
  - [ ] **低信頼/必須欠落** → Phase2は `FAILED` 扱い（将来 `needsReview` フラグ/状態で運用しやすいよう、情報はDBに保持）
- [ ] 方針: `categoryMinor` 不一致（許容値外/表記ゆれ）→ **低信頼要因**として扱う
- [x] 方針: OS通知タップ時の遷移先 → **対象レシートの詳細画面**
- [x] 方針: 通知履歴はPhase2最小では **永続化しない**
- [x] 方針: Gemini実装は **REST + OkHttp**

### 2.1 設定（APIキー）
- [x] 設定画面（最小）: APIキー入力/更新/削除
- [x] APIキーを安全保管（Keystore/EncryptedSharedPreferences等）
- [x] APIキー変更時の確認ダイアログ（フールセーフ）
- [x] 疎通テスト（テキストでOK → 画像へ拡張）

### 2.2 Gemini APIクライアント
- [x] 送信するリクエスト形式を確定（画像 + プロンプト + 厳格JSON指定）
- [x] HTTPクライアント導入（Ktor/OkHttpなど）とリトライ方針
- [x] タイムアウト/エラー分類（ネットワーク/認証/レート制限/パース失敗）

### 2.3 JSONスキーマ（厳格）
- [ ] `docs/REQUIREMENTS.md` のJSONスキーマに対応するデータクラス
- [ ] 厳格パース（不一致/欠落は低信頼へ）
- [x] `gemini_results`（生JSON）を保存する（要件準拠: 再現/デバッグ/将来の再解析のため）

### 2.4 解析キュー（WorkManager）
- [x] `analysis_queue` テーブル（状態/試行回数/エラー）
- [x] 送信確定時にキュー追加（重複防止）
- [x] Worker実装（1件ずつ取り出し → Gemini → DB更新）
- [x] 失敗時: backoff / 最大試行回数 / FAILED遷移
- [x] キューの可観測性: 設定/通知タブに「キュー件数」「最終エラー」を出す（最小）

### 2.x 端末内ファイルのメンテ（WorkManagerへ段階移行）
- [ ] 画像の40日掃除を WorkManager に移行（起動速度保護）
  - [ ] 端末アイドル/充電中などの条件は要検討（初期は毎日1回程度でもOK）
  - [ ] `receipt_images.deletedAt IS NULL` のみ対象にして冪等にする

### 2.5 通知（要件準拠の最小）
- [x] OS通知: 解析完了（DONE）
- [x] OS通知: 解析失敗（FAILED）
- [ ] 通知タブ（最小・運用ハブの土台）:
  - [ ] 解析完了の履歴（見逃し対策）
  - [ ] 要確認（低信頼）の入口（件数と一覧への導線だけでも良い）

## Phase 3: DB拡張と分析（要件の価値を出す）

> 目標: **レシート＋明細行（アイテム）主**で保存し、集計/分析UIを作る

### 3.1 DBスキーマ拡張（Room）
- [ ] `receipt_items`（明細行）追加
- [ ] `gemini_results`（生JSON）追加（Phase2で保存する前提で確定）
- [ ] Receiptの必須フィールド（merchantName/totalAmountYen/receiptDatetime等）を追加
- [ ] 調整行（isAdjustment=1）の自動生成ロジック
- [ ] 論理削除（deletedAt/deleteReason）追加
- [ ] カテゴリ固定リストをコード化（大カテゴリenum + 小カテゴリ日本語ラベルの固定配列/マップ）
- [ ] Gemini結果のバリデーション（表記ゆれ不可）:
  - [ ] `categoryMajor` がenum外 / `categoryMinor` が許容値に完全一致しない → needsReviewへ
  - [ ] 不一致時のフォールバック方針（例: `OTHER/その他`に落とす or 低信頼のみ）を確定
- [ ] レシートの必須/裁量バッジ算出ロジック（要件）:
  - [ ] 明細（`isAdjustment=0`）の金額加重平均で `necessityScore` を算出
  - [ ] 境界50で二値化（必須寄り / 裁量寄り）し一覧表示へ反映

### 3.2 低信頼（needsReview）フロー
- [ ] 低信頼判定（confidence < 0.7 / 必須欠落）
- [ ] 通知タブ（最小）: 要確認一覧
- [ ] 修正画面（最小）: 必須項目 + 支払手段 + 必須度スコア
- [ ] 必須欠落の定義を要件に合わせて固定（取引日時/合計/店名/カテゴリ大+小）
- [ ] 欠落時の確定方針を決める（例: needsReviewの間は集計から除外 or 別扱い）

### 3.3 一覧/詳細のアップグレード
- [ ] 一覧（レシート単位）: 日時/店名/合計/必須vs裁量バッジ
- [ ] 詳細（レシート）: 明細行表示（元順序）
- [ ] 詳細（レシート）: needsReview時のみ状態表示、必須欠落は強調表示
- [ ] 削除フロー（要件準拠）:
  - [ ] 論理削除 + 削除理由（`MISTAKE/REFUND/OCR_MISS/DUPLICATE/OTHER`）
  - [ ] 削除時は確認ダイアログ必須（理由選択を含む、デフォルトは`MISTAKE`）

### 3.4 分析タブ（初期）
- [ ] 必須 vs 裁量 比率（円/件数）
- [ ] 無駄遣い候補TOP（スコア×金額）
- [ ] 月切り替え
