# 実装計画（細分化）

この計画は、要件（`docs/REQUIREMENTS.md`）に沿って、実装順を

1. **最低限の動くUI**
2. **Geminiとの連携**
3. **DBと分析**
4. **バックアップ**

の順で進めるためのタスク分解です。

---

## Phase 0: 基盤（横断の前提）

### 0.1 リポジトリ運用
- [ ] ブランチ運用: `feat/*` → PR → `main`（通常マージ）
- [ ] `main`保護: PR必須 + `Android CI` 必須 + force push/delete禁止
- [ ] CIが落ちたときの一次切り分け手順を `docs/` に追記（必要になったら）

### 0.2 アーキテクチャの最小方針
- [ ] 画面（Compose）: `ui/*`
- [ ] データ（Room/Repository）: `data/*`
- [ ] 外部API（Gemini/Drive）: `integrations/*`（後で追加）
- [ ] 長時間処理（キュー/WorkManager）: `workers/*`（後で追加）

## Phase 1: 最低限の動くUI（MVP）

> ※ Gemini/通知/バックアップはまだ行わない（PENDING保存のみ）

- [ ] 初期は分析/通知/設定は「仮画面（TODO表示）」でOK
- [ ] 画面遷移を下タブ構造に統合（Camera/Preview/Listをタブ配下に置く）

### 1.1 カメラ（CameraX）
- [x] CameraX依存追加（camera2/lifecycle/view）
- [x] CAMERA権限の要求
- [x] プレビュー表示
- [x] シャッターでJPEG保存（アプリ内部ストレージ）
- [ ] 画像の保存命名規則を固定（例: `receipts/<uuid>.jpg`）※現状でOKならスキップ
- [ ] 保存時に **長辺1600pxへリサイズ + JPEG品質85** を適用（要件準拠）
- [ ] 画像メタ取得（width/height/byteSize）を保存後の実ファイルで確定させる
- [ ] 40日削除に備えて「capturedAt基準のretentionUntil」を保存（Phase1で持つかは任意）

### 1.2 送信前プレビュー（フールセーフ）

### 1.3 ローカル保存（未解析状態）

### 1.4 一覧画面（最小）

- [ ] レシート詳細画面（最小）: 画像のみ表示（解析結果は未実装でもOK）

### 1.4.1 画像の保持/削除（要件準拠の最小）

### 1.5 MVPの確認項目（実機）

## Phase 2: Geminiとの連携（解析を動かす）

> 目標: **送信確定 → 即キュー投入 → Gemini解析 → DBへ反映**  
> ※ 解析はバックグラウンドで順番に（要件の「1件ずつ」）

### 2.1 設定（APIキー）
- [ ] 設定画面（最小）: APIキー入力/更新/削除
- [ ] APIキーを安全保管（Keystore/EncryptedSharedPreferences等）
- [ ] APIキー変更時の確認ダイアログ（フールセーフ）
- [ ] 疎通テスト（テキストでOK → 画像へ拡張）

### 2.2 Gemini APIクライアント
- [ ] 送信するリクエスト形式を確定（画像 + プロンプト + 厳格JSON指定）
- [ ] HTTPクライアント導入（Ktor/OkHttpなど）とリトライ方針
- [ ] タイムアウト/エラー分類（ネットワーク/認証/レート制限/パース失敗）

### 2.3 JSONスキーマ（厳格）
- [ ] `docs/REQUIREMENTS.md` のJSONスキーマに対応するデータクラス
- [ ] 厳格パース（不一致/欠落は低信頼へ）
- [ ] `gemini_results`（生JSON）を保存する（要件準拠: 再現/デバッグ/将来の再解析のため）

### 2.4 解析キュー（WorkManager）
- [ ] `analysis_queue` テーブル（状態/試行回数/エラー）
- [ ] 送信確定時にキュー追加（重複防止）
- [ ] Worker実装（1件ずつ取り出し → Gemini → DB更新）
- [ ] 失敗時: backoff / 最大試行回数 / FAILED遷移
- [ ] キューの可観測性: 設定/通知タブに「キュー件数」「最終エラー」を出す（最小）

### 2.5 通知（要件準拠の最小）
- [ ] OS通知: 解析完了（DONE）
- [ ] OS通知: 解析失敗（FAILED）
- [ ] 通知タブ（最小・運用ハブの土台）:
  - [ ] 解析完了の履歴（見逃し対策）
  - [ ] 要確認（低信頼）の入口（件数と一覧への導線だけでも良い）

## Phase 3: DB拡張と分析（要件の価値を出す）

> 目標: **レシート＋明細行（アイテム）主**で保存し、集計/分析UIを作る

### 3.1 DBスキーマ拡張（Room）
- [ ] `receipt_items`（明細行）追加
- [ ] `gemini_results`（生JSON）追加（Phase2で保存する前提で確定）
- [ ] Receiptの必須フィールド（merchantName/totalAmountYen/receiptDatetime等）を追加
- [ ] 調整行（isAdjustment=1）の自動生成ロジック
- [ ] 論理削除（deletedAt/deleteReason）追加
- [ ] カテゴリ固定リストをコード化（大カテゴリenum + 小カテゴリ日本語ラベルの固定配列/マップ）
- [ ] Gemini結果のバリデーション（表記ゆれ不可）:
  - [ ] `categoryMajor` がenum外 / `categoryMinor` が許容値に完全一致しない → needsReviewへ
  - [ ] 不一致時のフォールバック方針（例: `OTHER/その他`に落とす or 低信頼のみ）を確定
- [ ] レシートの必須/裁量バッジ算出ロジック（要件）:
  - [ ] 明細（`isAdjustment=0`）の金額加重平均で `necessityScore` を算出
  - [ ] 境界50で二値化（必須寄り / 裁量寄り）し一覧表示へ反映

### 3.2 低信頼（needsReview）フロー
- [ ] 低信頼判定（confidence < 0.7 / 必須欠落）
- [ ] 通知タブ（最小）: 要確認一覧
- [ ] 修正画面（最小）: 必須項目 + 支払手段 + 必須度スコア
- [ ] 必須欠落の定義を要件に合わせて固定（取引日時/合計/店名/カテゴリ大+小）
- [ ] 欠落時の確定方針を決める（例: needsReviewの間は集計から除外 or 別扱い）

### 3.3 一覧/詳細のアップグレード
- [ ] 一覧（レシート単位）: 日時/店名/合計/必須vs裁量バッジ
- [ ] 詳細（レシート）: 明細行表示（元順序）
- [ ] 詳細（レシート）: needsReview時のみ状態表示、必須欠落は強調表示
- [ ] 削除フロー（要件準拠）:
  - [ ] 論理削除 + 削除理由（`MISTAKE/REFUND/OCR_MISS/DUPLICATE/OTHER`）
  - [ ] 削除時は確認ダイアログ必須（理由選択を含む、デフォルトは`MISTAKE`）

### 3.4 分析タブ（初期）
- [ ] 必須 vs 裁量 比率（円/件数）
- [ ] 無駄遣い候補TOP（スコア×金額）
- [ ] 月切り替え

## Phase 4: バックアップ（Google Drive / App folder）

> 目標: **日次自動バックアップ + 月初処理 + 復元/マージ** を要件どおりに

### 4.1 Drive連携の最小
- [ ] Google Sign-In（端末アカウント）で認証
- [ ] `appDataFolder` へファイル作成/更新/取得/削除
- [ ] スコープは `drive.appdata` のみ

### 4.2 エクスポート（JSON）
- [ ] `backupSchemaVersion=1.0` のJSON生成
- [ ] `current-month.json` 上書き（日次）
- [ ] バックアップ対象は取引データのみ（画像除外）

### 4.3 月初処理
- [ ] `archive-YYYY-MM.json` 生成（先々月）
- [ ] `full-YYYY-MM-01.json` 生成（全期間スナップショット）
- [ ] フルスナップショットは最新+1つ前のみ保持（古いもの削除）

### 4.4 復元/マージ
- [ ] 最新full → archive → current-month の順に適用
- [ ] `updatedAt` 優先でマージ
- [ ] 破損時のフォールバック（fullを1つ前へ）

### 4.4.1 復元UX（手動）
- [ ] 設定画面に「復元」ボタンを設置
- [ ] 実行前に確認ダイアログ（上書き/マージ方針の注意を表示）
- [ ] 復元成功/失敗を通知タブに記録

### 4.5 バックアップログ（通知タブ/設定タブ）
- [ ] 成功/失敗ログを表示
- [ ] 失敗時の再試行導線（将来でも可）
