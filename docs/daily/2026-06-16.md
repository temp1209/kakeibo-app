# 日報 2026-06-16

## 概要

Phase 7（実利用フィードバック対応）のうち **7.3 APIキー送信ガード** と **7.2 Drive バックアップ** を中心に実装した。Drive は 403 → 404 の切り分けを経てバックアップ成功まで到達したが、復元テストで **データ保全まわりの設計欠陥**（空DB上書き・当月エクスポート漏れ）が顕在化した。夕方に **manifest + 不変スナップショット（v2）** へ全面移行し、安全ガードを多層化した。**本日の最終版は実機未確認**（ビルド・コンパイルのみ）。

### 2. Phase 7.2 — Drive バックアップ（段階的修正）

| 段階 | 症状 | 対応 |
|------|------|------|
| 初回 | 403 `insufficientFilePermissions` | `appDataFolder` 親指定漏れを修正（create 時 `parents: ["appDataFolder"]`） |
| 次 | 404 on `uploadType=media` | 新規作成を multipart 一括、更新を PATCH に変更 |
| 再ログイン時 | 404（古い fileId への上書き） | 404 時は新規作成 → 成功後に重複削除（削除先行を廃止） |
| UX | 生 HTML が Snackbar に表示 | `DriveBackupUserMessages` で短文化、再ログイン導線 |

- `GoogleSignInHelper.hasDriveAppDataScope()` 追加
- ログイン成功時の自動バックアップは後述の事故対応で **廃止**
- SHA-1 切り分け: `keytool` が PATH に無い問題を確認。Android Studio JBR からの実行手順を整理（`EXTERNAL_SETUP.md` §6 と整合）

### 3. 実機確認で判明したバックアップの欠陥

| 問題 | 原因 | 対応 |
|------|------|------|
| 新規レシートが復元されない | `current-month` の当月判定が印字日のみ。解析後に先月印字日になるとエクスポート漏れ | `capturedAt` / `updatedAt` も当月判定に含める |
| アプリデータ削除 → ログインで Drive データ消失 | 空 DB で自動バックアップが走り JSON を上書き | ローカル0件 + Drive 既存時はバックアップ拒否、ログイン時は接続のみ |
| 件数が減った状態での上書きリスク | ガードなし | `BackupSafetyGuard` + `BackupRegressionException`（manifest / 旧方式の件数推定） |

- 復元テスト結果（参考）: 同一端末マージ「反映0・スキップ8」、データ削除後「反映5・スキップ3」。論理削除分は復元後も一覧に出ない（仕様どおり）

### 4. Drive バックアップ v2 — manifest + 不変スナップショット（夕方）

旧形式（`current-month.json` 等の上書き）を廃止し、**既存 Drive 構成との互換は考慮しない**（テストデータのみ、実データは PC 退避）。

| ファイル | 役割 |
|----------|------|
| `manifest.json` | 唯一の可変ファイル（最新スナップショットへのポインタ + 件数メタ） |
| `snap-{日時}-CURRENT_MONTH.json` | 当月分（毎回新規作成） |
| `snap-{日時}-FULL_SNAPSHOT.json` | フル（月初に新規、manifest で最新2世代保持） |
| `snap-{日時}-ARCHIVE_MONTH-{YM}.json` | 月次アーカイブ |

- 新規: `BackupManifest.kt`, `ManifestCodec`, `SnapshotFileNames`
- 刷新: `DriveBackupRepository.kt`, `DriveBackupOrchestrator.kt`, `BackupSafetyGuard.kt`
- 復元: manifest 参照のみ。ダウンロード失敗は黙殺せずエラー
- prune: manifest 未参照の `snap-*` と旧形式ファイルを削除
