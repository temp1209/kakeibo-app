## 1. 背景と課題

### 現状

- `necessityScore`（0〜100）は **Gemini 解析プロンプト**で付与される（`ReceiptAnalysisPrompt` → `AnalysisWorker`）
- 集計の二値化境界 **50** はアプリ側固定（`REQUIREMENTS.md` §7）
- 「一般家庭向け」の価値観が **全ユーザー共通**であり、人によって異なる「必須／裁量」の感覚を反映できない

### 課題

| 課題 | 内容 |
|------|------|
| 主観の差 | 節約重視・コンビニ削減・子育て世帯など、必須の範囲が人によって違う |
| 例の直渡しは不適 | 解析のたびに訂正例リストをプロンプトに載せると、**呼び出しごとに解釈がブレる** |
| API コスト | 毎回の長いプロンプトはトークン増・不安定化の原因 |

### 方針（合意済み）

- **自由記述の長文設定はしない**
- **目的プリセット**（1つ）+ **訂正例**（運用で追加）を入力とする
- 訂正例は解析に **直接渡さない**。設定保存時に **方針コンパイル AI** が **客観的な判定基準**へ言語化し、端末に保存する
- 解析・再スコアは **コンパイル済み基準のみ**を参照する（**ハイブリッド**: プリセット土台はアプリ内テンプレ、ユーザー固有部分のみ AI が生成）

## 2. 目標と完了条件

### 目標

- ユーザーごとに **必須／裁量の基本方針**をアプリ内で設定できる
- 設定は **一貫した客観基準**として解析に反映され、呼び出しごとのブレを抑える
- API 使用量を抑える（コンパイルは **設定変更時のみ**）
- 方針変更後、**当月分の明細**を任意で再スコアできる

### 完了条件（Phase 9）

- [x] 設定タブで目的プリセットを選び、保存＝コンパイルできる
- [x] コンパイル結果の **1行要約**と **全文表示**ができる
- [x] 新規レシート解析にコンパイル済み基準が注入される
- [x] 明細から訂正例を登録でき、未コンパイル時は **未反映**と分かる（案1）
- [x] 方針変更後、**当月分**の再スコアを **1ジョブ**で実行できる（明細数上限で API 分割）
- [x] 手動 JSON バックアップに **ユーザー固有設定**が含まれる
- [x] 実機でプリセット変更 → 解析 → 再スコアの一連が動作する

### スコープ外（初版）

- ユーザーによるスコア帯（数値）の直接編集
- カテゴリマスタの全面カスタマイズ
- 過去全期間の一括再スコア（当月のみ）
- 訂正例追加のたびの自動コンパイル（設定保存時のみ）
- 画像の再読み取り（再スコアは既存明細データのみ）

## 3. 概念モデル

### 3.1 三つの API ライン

```
┌──────────────────────────────────────────────────────────┐
│ ライン① 方針コンパイル（設定「保存」時のみ）                 │
│  入力: purposeId + 訂正例ストア + プリセット土台テンプレ      │
│  出力: compiledPolicy（機械向け原文）+ userSummary（1行）   │
└──────────────────────────────────────────────────────────┘
                           ↓ 端末保存（バージョン付き）
┌──────────────────────────────────────────────────────────┐
│ ライン② レシート解析（撮影フロー・現行どおり）               │
│  入力: 画像 + 固定ベースルール + compiledPolicy.promptBlock │
│  出力: 明細 + necessityScore                              │
└──────────────────────────────────────────────────────────┘
                           ↓ 任意（コンパイル後）
┌──────────────────────────────────────────────────────────┐
│ ライン③ 必須度再スコア（当月・1ジョブ）                     │
│  入力: 当月明細（itemName, category, lineTotalYen 等）      │
│        + compiledPolicy.promptBlock                       │
│  出力: 更新 necessityScore → DB 更新                      │
└──────────────────────────────────────────────────────────┘
```

| ライン | トリガー | API 回数の目安 |
|--------|----------|----------------|
| ① コンパイル | 設定画面で「保存」 | **1回／保存操作** |
| ② 解析 | レシート送信 | 現行どおり（1画像1回） |
| ③ 再スコア | ユーザーが当月再スコアを実行 | **⌈当月明細数 ÷ 上限⌉ 回** |

### 3.2 ハイブリッド構成

| 層 | 生成主体 | 内容 |
|----|----------|------|
| **ベースルール** | コード（`ReceiptAnalysisPrompt` の OCR/分類部分） | スキーマ・confidence・カテゴリ許可ペア等。全ユーザー共通 |
| **プリセット土台** | **`NecessityPresetTemplates`**（`data/prompt/necessity/`） | `purposeId` ごとの方針文・**スコア帯**・**境界ケース表** |
| **ユーザー固有ルール** | **コンパイル AI**（`NecessityCompilePrompt`） | 訂正例ストアを一般化した追記ルール（`userRulesBlock`） |
| **マージ結果** | **`NecessityPolicyPromptMerger`** | `promptBlock` = 土台 + ユーザー固有 + プリセット別の調整幅注釈 |

**フル AI コンパイル（採用しない）**: プリセットの意味まで AI が毎回文章化 → 同じ選択でもブレうる。

## 4. 機能要件

### 4.1 目的プリセット（1つ選択）

自由記述ではなく、固定候補から **ラジオ／カードで1つ**選ぶ。

| ID | 表示名（案） | テンプレの趣旨 |
|----|-------------|----------------|
| `BALANCE` | バランス（デフォルト） | 現行プロンプトに近い中立 |
| `SAVE` | 節約・無駄遣いを減らす | 嗜好品・衝動買いを裁量寄りに |
| `MANAGE` | 家計を把握する | 分類の一貫性重視、境界は中立 |
| `CONVENIENCE_CUT` | コンビニ・外食を減らしたい | コンビニ嗜好品・カフェ単体を裁量寄りに |
| `MINIMAL` | 必要最低限だけ必須 | 必須帯を狭く、グレーを裁量寄りに |

- 初版は上記5種。文言は UX で微調整可
- **プリセットごとにスコア帯・境界ケース表を差し替え**（`SAVE` / `MINIMAL` は嗜好品を低め、`CONVENIENCE_CUT` はコンビニ嗜好品を重点的に下げる等）
- 各プリセットに **初期注目チップ**（任意オン）を紐づけてもよい（コンパイル入力の補助。解析には載せない）— **未実装**

### 4.2 訂正例ストア

ユーザーが「自分にとっての必須／裁量」の具体を登録する。**コンパイルの入力**であり、解析プロンプトには **直接含めない**。

#### 登録方法

| 経路 | 操作 |
|------|------|
| レシート修正画面 | 必須度スライダーを変更して保存 → 変更行を自動で訂正例に追加（確認ダイアログなし） |
| レシート修正画面 | 「必須度をまとめて上げる／下げる」で一括調整（5 単位） |

#### 1件のデータ

```json
{
  "correctionId": "uuid",
  "phrase": "ポテチ",
  "scoreBefore": 45,
  "scoreAfter": 15,
  "receiptId": "uuid",
  "merchantName": "セブンイレブン",
  "sourceItemName": "ポテトチップス　うすしお",
  "createdAt": "2026-07-12T12:00:00+09:00"
}
```

| フィールド | 説明 |
|------------|------|
| `phrase` | 短い代表フレーズ（商品名から正規化） |
| `scoreBefore` / `scoreAfter` | AI 付与値 → ユーザー希望値（0〜100） |
| `receiptId` | 登録元レシート（設定画面でレシート単位表示） |
| `merchantName` | 店名（コンパイル AI に文脈として渡す） |
| `sourceItemName` | 登録元の明細名（トレース用） |

- 上限: **最大20件**（超過時は古いものから削除）
- 同一 `receiptId` + `phrase` は最新で上書き
- コンパイル確認 OK 後は訂正例キューを **すべてクリア**（方針に取り込み済み）

### 4.3 コンパイル（ライン①）

#### タイミング

- **設定画面で「方針を保存」したときのみ**
- 訂正例の追加・削除だけでは **自動コンパイルしない**（API 削減・「一度合えばあまり触らない」想定）

#### 未反映の扱い（案1・合意済み）

- 訂正例を追加／削除したがまだ保存していない場合:
  - 設定に **「未反映の訂正があります」** を表示
  - **解析・再スコアは直前のコンパイル済み方針**を使用
- 保存成功後にコンパイル実行 → 新方針が有効化

#### コンパイル AI の入出力

**入力**

- `purposeId` → アプリが展開した **プリセット土台テンプレ**
- `corrections[]` → 訂正例ストア（**レシート／店名ごとにグループ化**して渡す）
- 既存のコンパイル済み `userSummary` + `userRulesBlock`（改善・統合のたたき台）
- 訂正例ごとに AI が **なぜ元スコアが合わなかったか** を推論し、方針を更新

**出力（厳格 JSON）**

```json
{
  "policyVersion": 3,
  "purposeId": "SAVE",
  "compiledAt": "2026-07-12T16:00:00+09:00",
  "userSummary": "嗜好品とコンビニ菓子を裁量寄りに判定します。",
  "presetBlock": "（アプリ展開・保存用コピー）",
  "userRulesBlock": "（AI が訂正例から一般化した追記）",
  "promptBlock": "（presetBlock + userRulesBlock をマージした解析用全文）"
}
```

| フィールド | 用途 |
|------------|------|
| `userSummary` | 設定画面に常時表示する **1行要約**（人間向け） |
| `promptBlock` | ライン②③ に注入する **機械向け原文** |
| `userRulesBlock` | デバッグ・全文表示用。訂正例の一般化結果 |

- コンパイル失敗時: エラー表示、**前回の有効な `compiledPolicy` を維持**
- `policyVersion` は保存のたびにインクリメント

### 4.4 設定 UI

**配置**: 設定タブ「必須度の方針」セクション

| 要素 | 要件 |
|------|------|
| 目的プリセット | 5択（**ドロップダウン**。変更頻度は低いが占有面積を抑える） |
| 現在の方針 | `userSummary` を1行表示 |
| 全文 | 「判定基準の全文を見る」ボタン → ダイアログ／BottomSheet で `promptBlock` 表示 |
| 訂正例一覧 | **レシート（店名）単位**でグループ表示・行ごとに削除 |
| 未反映バナー | 訂正例に未コンパイルの差分があるとき表示 |
| 保存 | 「方針を保存してコンパイル」→ プレビュー → **確認ダイアログ** → OK で確定＋訂正例クリア |
| 再スコア | 確定後、「今月の明細を新しい方針で再計算する」（任意） |

**スコープ外**: ユーザーが `promptBlock` を直接編集する UI

**設定タブ（APIキー・未実装）**: キー保存済み時は入力欄を隠し「変更する」で展開（[`REQUIREMENTS.md`](../REQUIREMENTS.md) §13 参照）

### 4.5 レシート解析（ライン②）

- 現行フローを維持（撮影体験を最優先）
- `ReceiptAnalysisPrompt` が OCR 部 + `NecessityScorePromptSections`（`promptBlock` + プリセット別スコア帯・境界ケース）を組み立て、`AnalysisWorker` が呼び出す
- 未コンパイル時は **`NecessityPolicyPromptMerger.fallbackPolicy(purposeId)`**（選択プリセットの土台のみ）
- **訂正例リストは渡さない**

### 4.6 当月再スコア（ライン③）

#### トリガー

- コンパイル成功後の確認ダイアログ、または設定内の明示ボタン
- **任意**（自動実行しない）

#### 対象範囲

- **カレンダー月（JST）の当月**
- 対象レシート: `receiptDatetime` が当月、なければ `capturedAt` が当月
- 対象明細: 上記レシートに紐づく `receipt_items`（`isAdjustment=0`）
- **画像は再送信しない**

#### ジョブ単位

- ユーザー操作 **1回 = アプリ内 1ジョブ**（`RescoreWorker` 等）
- 進捗表示（例: 「再計算中 45/120 件」）

#### API 分割

- 1リクエストあたりの明細数に **上限**を設ける（初版定数: **`MAX_ITEMS_PER_RESCORE_REQUEST = 50`** — 実装時にトークン計測で調整可）
- 当月明細を上限ごとにチャンク分割し、順次ライン③を呼ぶ
- 各レスポンスで `itemId`（または `receiptId`+`lineIndex`）と新 `necessityScore` を返し DB 更新
- レシート単位の加重平均・一覧バッジは既存ロジックで再計算

#### 再スコア専用プロンプト

- 実装: `NecessityRescorePrompt`（`data/prompt/necessity/`）
- 入力: 明細リスト（JSON）+ `compiledPolicy.promptBlock` + **`NecessityScorePromptSections`**（解析と同じスコア帯・境界ケース）
- 出力: 厳格 JSON（`[{ "itemId", "necessityScore" }, ...]`）
- **解析（OCR）プロンプトとは分離**した軽量プロンプト

### 4.7 バックアップ

手動 JSON エクスポートに **ユーザー固有設定**を含める（`backupSchemaVersion` を **1.2** に bump 想定）。

**追加ペイロード（案）**

```json
{
  "necessityPolicy": {
    "purposeId": "SAVE",
    "corrections": [ "..." ],
    "compiledPolicy": { "policyVersion": 3, "userSummary": "...", "promptBlock": "...", "compiledAt": "..." }
  }
}
```

| 項目 | インポート時 |
|------|-------------|
| `purposeId` + `corrections` | 復元。`compiledPolicy` があればそのまま使用 |
| `compiledPolicy` 欠落 | インポート後に **コンパイルを促す**（または `BALANCE` 土台のみで暫定運用） |

- 取引データとのマージ方針: 既存の `updatedAt` 優先ルールに **設定は端末ローカル優先 or 新しい export 優先** を実装計画で決定（推奨: **export のタイムスタンプが新しければ上書き**）

## 5. 非機能要件

| 項目 | 要件 |
|------|------|
| API コスト | コンパイルは保存時のみ。解析プロンプト増分は `promptBlock` 固定長に抑える |
| 一貫性 | 同一 `compiledPolicy` バージョンでは同一明細に近いスコア（完全保証はしない） |
| 失敗時 | コンパイル失敗 → 旧方針維持。再スコア失敗 → 部分完了を表示しリトライ可 |
| 撮影 UX | ライン②のレイテンシを悪化させない（`promptBlock` サイズ上限を設ける — 例: 2000文字） |
| セキュリティ | 方針は端末内保存。API 送信は Gemini へ（既存と同様） |

## 8. テスト手順（実機）

### コンパイル・解析

1. 設定で「節約」プリセットを選び保存 → 要約表示・全文表示を確認
2. レシート撮影 → 嗜好品のスコアがバランス時より低めになること（主観）
3. 訂正例を明細から追加 → 未反映バナー → 保存でコンパイル → バナー消滅

### 再スコア

1. 当月に複数レシートがある状態で方針変更・コンパイル
2. 「今月を再計算」→ 進捗表示 → 一覧バッジ・分析タブが更新されること
3. 51件以上の当月明細で API が分割されること（ログ or 進捗で確認）

### バックアップ

1. エクスポート JSON に `necessityPolicy` が含まれること
2. 削除 → インポート → 設定・要約が復元されること

### 回帰

1. 撮影 → 解析 → 一覧（コアフロー退行なし）
2. コンパイル未実施でも解析が動くこと（フォールバック）

## 9. リスク

| リスク | 対策 |
|--------|------|
| `promptBlock` が長すぎて解析が遅い／高い | 文字数上限・コンパイルプロンプトで簡潔化を指示 |
| コンパイル AI の出力が不安定 | JSON スキーマ厳格化・失敗時は旧版維持 |
| 再スコアが当月でも件数多 | 50件/リクエスト分割・バックグラウンド Worker |
| プリセットテンプレの保守 | `data/prompt/necessity/NecessityPresetTemplates.kt` で一元管理（`scoreBands` / `boundaryCases`） |

## 11. プロンプト構成（実装）

AI 向けプロンプトは **`data/prompt/`** に集約する。ビジネスロジック（コンパイル・Worker・Store）は `data/necessity/` / `data/analysis/` に残す。

```
data/prompt/
├── ReceiptAnalysisPrompt.kt          # ライン②（OCR + 必須度）
└── necessity/
    ├── NecessityCompilePrompt.kt     # ライン①
    ├── NecessityRescorePrompt.kt     # ライン③
    ├── NecessityScorePromptSections.kt  # 必須度セクション共通
    ├── NecessityPresetTemplates.kt   # プリセット土台・スコア帯・境界ケース
    └── NecessityPolicyPromptMerger.kt   # promptBlock マージ
```

### プリセット別チューニングの入口

| 関数 | 用途 |
|------|------|
| `presetBlock(purposeId)` | 方針文（コンパイル土台・マージ先頭） |
| `scoreBands(purposeId)` | スコア帯の目安（解析・再スコアに注入） |
| `boundaryCases(purposeId)` | 境界ケース具体例表（9行） |
| `defaultSummary(purposeId)` | 未コンパイル時の1行要約 |

**調整幅注釈**（マージ末尾）は `NecessityPolicyPromptMerger` でプリセット別:

- `BALANCE` / `MANAGE`: ±10点程度
- `SAVE` / `MINIMAL`: 15〜25点の帯シフト可
- `CONVENIENCE_CUT`: コンビニ嗜好品を10〜20点下げ優先

### コンパイルプロンプトの訂正例表現

`NecessityCompilePrompt` は訂正例の `scoreBefore → scoreAfter` 差分から **調整の強さ**（大幅に下げ／やや下げ／微調整等）を注釈し、コンパイル AI が帯シフトの意図を拾いやすくする。`userRulesBlock` 自体は数値スコアを書かないルールのまま。

### 既知の改善余地

| 項目 | 内容 |
|------|------|
| 再スコアに店名なし | 明細 JSON に `merchantName` を含めていない |
| プリセット変更のみ | コンパイル済み `promptBlock` は自動再生成されない（保存コンパイルが必要） |
| 境界ケース数値 | 実利用の訂正例を見ながら `SAVE` / `MINIMAL` の帯を継続チューニング |
