# 家計簿アプリ 要件定義（詳細化メモ）

## 目的
- Android向けの家計簿アプリを自作する。
- レシート撮影 → Gemini APIへ送信 → 内容抽出/分類 → 保存 → 集計/一覧表示までを行えること（解析時間は許容し、撮影体験を最優先）。
- 無駄遣い（裁量的支出）を可視化できること。
- 端末のデータをバックアップできること。

## 実装サマリ（初期リリースの核）
- **起動=即カメラ** → 撮影 → **送信前プレビューで「送信確定」** → キュー投入 → バックグラウンドで解析（1件ずつ） → 完了通知 → 原則自動確定（低信頼のみ要確認）
- **明細行（アイテム）主**で保存・分類（カテゴリ/必須度/信頼度はアイテム単位）
- **一覧はレシート単位**で一覧性重視、探索/深掘りは分析タブへ
- **手動 JSON バックアップ**（設定からエクスポート/インポート）+ 月次リマインド（一覧タブ）

## 現時点で確定している要件（ユーザー提示）
- **対象プラットフォーム**: Android
- **入力**: カメラでレシート等を撮影
- **解析**: 撮影画像をAPIでGeminiへ送信し、内容の解析や分類まで行う
- **表示/集計**: 分析された内容の集計、一覧
- **体験**: 素早い起動
- **保全**: データのバックアップ

## 方針（確定・構造化）

### 1. スコープ/前提
- **対象**: Android（ネイティブ）
- **利用形態**: 端末1人用（ログインなし）
- **初期スコープ**: 支出のみ（レシート中心）
- **通貨/金額**: JPY固定・整数（円）
- **メモ欄**: 初期は不要
- **オフライン**: 初期は未対応（オンライン必須）。将来対応

### 2. 技術スタック
- **技術スタック（Android）**: Kotlin + Jetpack Compose

### 3. 入力/フロー（UX）
- **最重要速度要件**: 起動後、即時に撮影へ入れる（レシート受領→即撮影→廃棄の流れを止めない）
- **初回起動**: §14 のオンボーディング Wizard を表示。完了後は下記「通常起動」に入る
- **通常起動（2回目以降）**: ランチャー・前面復帰時は **カメラタブ**を表示（Phase 6.1）。通知 deep link 時のみ詳細へ直行
- **送信前確認（フールセーフ）**: 撮影後、送信前に画像プレビューを表示し、ユーザーが「送信確定」ボタンを押した場合のみGeminiへ送信する（解析キュー投入も確定後のみ）
  - プレビュー画面の操作（初期）: 送信確定 / キャンセル（=撮影へ戻る、撮り直し同等）
  - 回転/トリミング（初期）: 入れない
- **解析の非同期性**: 解析は時間がかかっても良い（撮影をブロックしない）
- **解析キュー**: 1件ずつ順番に解析（初期）
- **解析完了通知**: OS通知を出す
- **確定方針**: 解析結果は原則自動確定し一覧/集計へ反映。低信頼時のみ確認を挟む
- **ナビゲーション（初期）**: 下タブ5つ（一覧 / 分析 / カメラ / 通知 / 設定）。カメラは中央タブで、撮影時はシャッターボタン的UIも検討
- **送信確定後の挙動（初期）**: 送信確定（キュー投入）後は即カメラへ戻る
- **通知タブ（初期）**:
  - 解析完了の履歴（OS通知の見逃し対策）
  - 低信頼で要確認の一覧（修正画面への入口）
  - 「その他」日次まとめ（件数/該当レシートへのリンク）
  - ~~バックアップ結果（成功/失敗ログ）~~ → 未実装（手動バックアップは設定タブ。通知タブへのログは将来検討）
- **通知バッジ（初期）**: 要確認（低信頼）件数のみカウント
- **通知タブの役割（初期）**: ユーザーに働きかける（要対応の提示・導線を集約）

### 4. データモデル（支出）
- **一覧の基本単位**: レシート単位（店舗×日時×合計）を上位にし、配下に明細行を保持
- **一覧のデフォルト並び**: レシート日時（`receiptDatetime`）の降順（最新が上）
- **取引日時の優先順位**: レシート記載日時を最優先（取れない場合のみ撮影日時）
- **店舗情報**: 店名のみ保存
- **値引き/クーポン**: 考慮しない（実支払金額のみ記録）
- **返金/返品**: 取引の削除で対応
- **削除**: 論理削除 + 削除理由（選択式。理由は `MISTAKE/REFUND/OCR_MISS/DUPLICATE/OTHER`。デフォルトは `MISTAKE`）
- **削除（UI/運用）**:
  - 削除は確認ダイアログ必須（理由選択を含む）
  - 理由のデフォルトは「誤送信（MISTAKE）」とする
- **一覧タブ（初期）**:
  - 表示（1行）: レシート日時, 店名, 合計金額, 必須/裁量（バッジ）
  - フィルタ: 月, 大カテゴリ
  - 役割: 一覧性重視（探索/深掘りは分析タブの役割）
  - 必須/裁量バッジの算出: 明細行（`isAdjustment=0`）の **金額加重平均**で `necessityScore` を算出（\(\sum(lineTotalYen \times necessityScore) / \sum(lineTotalYen)\)）。二値化境界（50）で表示を決める

- **要確認（低信頼）導線（初期）**: 通知タブの「要確認一覧」から対象レシート/修正画面へ遷移できること
- **レシート詳細画面（初期）**:
  - 一覧タブの1行をタップして開く別画面
  - ヘッダ表示: レシート日時 / 店名 / 合計金額
  - 解析状態の表示: **要確認（低信頼）の時のみ**表示する
  - 低信頼時の入力促進: ユーザー入力が必要な項目（必須欠落など）は赤枠等で強調して入力を促す。問題ない場合は通常表示
  - 操作メニュー: 画面内にメニュー（例: ハンバーガー/三点メニュー）を用意し、削除・修正・画像確認などを格納する

### 5. 明細行（アイテム）
- **保存単位**: 明細行（商品ごと）
- **レシート詳細の明細並び（初期）**: 元の順序（レシートに近い並びを維持）
- **明細行の金額（保存）**: 行合計を保存
- **数量が取れない場合**: 数量=1
- **商品名が取れない場合**: 商品名="不明"として保存（低信頼寄り）
- **合計不一致（レシート合計 vs 明細合計）**:
  - レシート合計を正とする
  - 差分は「調整」行を自動追加（カテゴリ=その他）
  - 「調整」行は表（一覧/明細表示）に出さず、集計のみで用いる

### 6. カテゴリ（固定・2階層）
- **カテゴリ方針**: 初期に網羅的に定義し、ユーザーによる追加/変更/削除は不可
- **階層**: 大カテゴリ→小カテゴリ（2階層固定）
- **その他カテゴリ**: 許可（「その他」件数は日次まとめ通知）
- **日用品の分類軸**: モノの種類（アイテム種別）で分類（店種別では分類しない）
- **大カテゴリ（固定）**: 食費/日用品/交通/住居/光熱/通信/医療/教育/交際/娯楽/衣服/その他
- **小カテゴリ（固定）**:
  - 食費: スーパー, 外食, その他
  - 日用品: 衛生用品, 雑貨（小物）, 家具家電, その他
  - 交通: 電車/バス, 定期券, その他
  - 住居: 家賃/住宅ローン, 家具（大物）, その他
  - 光熱: 電気, ガス, 水道, その他
  - 通信: 携帯, インターネット, サブスク（通信系）, その他
  - 医療: 病院, 薬, 健康/検診, その他
  - 教育: 教材, その他
  - 交際: 交際費, 飲み会, 贈り物, 冠婚葬祭, その他
  - 娯楽: 趣味, 旅行, イベント, ゲーム（買い切り）, ゲーム（課金）, その他
  - 衣服: 衣類, 靴/バッグ, クリーニング, その他
  - その他: 返済, その他

### 7. 必須度（無駄遣い可視化）
- **目的**: 無駄遣い（裁量的支出）を可視化
- **スコア**: 0〜100（連続値）
- **付与単位**: 明細行（アイテム単位）
- **二値化境界（集計用）**: 50（\(\ge 50\)=必須寄り、\<50=娯楽/裁量寄り）
- **可視化/集計**:
  - 一覧でバッジ/色/ラベル表示
  - 月次の「必須 vs 裁量」比率（円/件数）
  - 無駄遣い候補TOP（裁量寄り × 金額大）
  - 裁量支出の閾値超え通知/警告（将来でも可）
- **分析タブ（初期）**: ダッシュボード（必須vs裁量比率 + 無駄遣い候補TOP を同一画面に配置）
- **検索機能（初期）**: 後回し（必要性が出たら検討）
- **無駄遣い候補TOPの期間（初期）**: 当月をデフォルト。月切り替えで過去月も閲覧できる
- **無駄遣い候補TOPの並び（初期）**:
  - デフォルト: スコア \((100 - necessityScore) \times lineTotalYen\) の降順
  - 切り替え: 金額（`lineTotalYen`）降順にも切り替え可能

### 8. 支払手段
- **支払手段（分類・固定）**: 現金, クレジットカード, デビットカード, 交通系IC, 電子マネー, QR決済, 振込/銀行引落, 不明
- **支払サービス名**: 自由テキスト（例: QR決済 + PayPay）

### 9. 返済（その他カテゴリ）
- **返済（内訳・固定）**: リボ, 奨学金, その他
- **返済の入力（初期）**: 手入力のみ
- **返済の支払手段**: 振込/銀行引落で固定
- **二重計上回避（クレカ/返済）**:
  - 原則: 購入時に支出として計上（クレカ利用も購入時に支出）
  - 返済: 支出に含める（家計キャッシュアウトとして可視化）
  - 注意: 同一支出を「購入時」と「返済時」に二重で登録しない（運用ルール）

### 10. Gemini連携
- **APIキー**: 初回起動で入力し端末に安全保管（Android Keystore等）
- **APIキー変更（初期）**: 設定から変更できる。ただし変更時は確認画面を出す（フールセーフ）
- **呼び出し**: アプリから直接呼び出す
- **モデル（初期）**: Gemini 2.5 Flash
- **出力形式**: 厳格JSON（スキーマ固定）必須（パース失敗は再試行/低信頼扱い）
- **主出力**: 明細行（アイテム単位）を主として抽出・分類
- **カテゴリ/必須度の付与主体**: Geminiが各明細行に直接付与（低信頼時のみ修正）
- **個人情報配慮（初期）**: 画像はそのまま送信（動くもの優先）

### 11. 低信頼（確認/修正フロー）
- **低信頼判定**: A（confidence < 0.7）+B（必須フィールド欠落）
- **必須フィールド**: 取引日時, 合計金額, 店名, カテゴリ（大+小）
- **confidenceの粒度**: 明細行（アイテム）ごとに保持
- **低信頼時の確認/修正**:
  - 必須フィールドが欠落している場合、ユーザーが入力して補完（補完できない限り確定しない）
  - 支払手段/必須度スコアは常に修正可能
  - 初期は編集範囲を絞る（実装後に精度が悪ければ緩和）:
    - 低信頼修正画面では「必須項目 + 支払手段 + 必須度」を主に扱い、その他の自由編集は原則行わない
    - 必須度スコア編集: 明細行ごとに編集（個別に直す）
    - カテゴリ編集: 欠落している場合のみ選択入力で補完できる（誤分類の修正は初期はしない）
  - 明細行は「全く読み取れなかった」場合のみ手入力を許可
  - 「全く読み取れなかった」判定: 明細0件、または商品名がほぼ不明な行しかない
  - 明細手入力の項目: 商品名 + 金額 + 数量

### 12. バックアップ/保全

> **2026-07-11 改定**: Google Drive 自動バックアップは廃止。手動 JSON + 月次リマインドに集約。旧 Drive 要件は [`archive/REQUIREMENTS_DRIVE_BACKUP.md`](archive/REQUIREMENTS_DRIVE_BACKUP.md)。

- **方式**: 設定タブから **JSON エクスポート / インポート**（Storage Access Framework）。ユーザーが保存先を選ぶ
- **月次リマインド**: 一覧タブ表示時、今月まだエクスポートしていなければダイアログ（最大月1回）。「今すぐバックアップ」でエクスポートへ直行
- **暗号化（初期）**: しない（将来検討）
- **バックアップ範囲（初期）**: 取引データのみ（レシート画像・解析キューは含めない）
- **レシート画像の端末内保持（初期）**: 40日保持して自動削除（月次集計 + バッファ）
- **レシート画像の保存方式（端末内・初期）**:
  - 保存場所: アプリ内部ストレージ（他アプリから見えない）
  - 形式/圧縮: JPEG品質85 + 長辺1600pxにリサイズ
  - 削除タイミング: アプリ起動時に40日超過分を掃除（冪等）
  - 表示方針: 通常は表示しない。低信頼時の確認など「必要なときだけ」読み込んで表示できる
- **エクスポート種別（現行）**: `FULL_SNAPSHOT`（全期間の単一 JSON）。ファイル名例: `kakeibo-backup-YYYY-MM-DD.json`
- **復元方針（初期）**: マージ（新しい更新を優先して統合）
- **マージ優先（初期）**: `updatedAt` が新しい方を採用
- **安全ガード（簡易）**:
  - ローカル 0 件でのエクスポートは拒否
  - インポート前に件数・日時を確認するダイアログ
  - ローカルにデータがある状態でバックアップ件数が大幅に多い場合は追加確認

#### バックアップ JSON スキーマ（確定版）

手動エクスポート/インポートの JSON スキーマ。**復元/マージ実装の基準**として定義する。実装: `FileBackupOrchestrator`, `BackupMerge`。

##### エクスポート種別（現行）

- **フルスナップショット**: `exportType: "FULL_SNAPSHOT"`
  - 対象: 全期間（アクティブな取引データ）
  - ユーザーが SAF で保存先を指定（Drive 等のクラウドにも手動で置ける）

> 旧 Drive 運用時の `CURRENT_MONTH` / `ARCHIVE_MONTH` 分割は廃止。スキーマ互換のため `exportType` 値は残す。

##### ルート（必須）
- `backupSchemaVersion`: `"1.0"`
- `exportType`: `"CURRENT_MONTH"` / `"ARCHIVE_MONTH"` / `"FULL_SNAPSHOT"`
- `exportedAt`: ISO 8601
- `rangeStart` / `rangeEnd`: ISO 8601（このファイルがカバーする期間。FULLは全期間でも値は入れる）
- `app`: `{ "packageName": string, "versionName": string, "versionCode": number }`
- `data`: エクスポート本体

##### `data`（必須）
初期は**画像は含めない**（端末内保持のみ）。解析キュー等の実行時情報も含めない。
- `receipts`: `ReceiptBackup[]`
- `receiptItems`: `ReceiptItemBackup[]`

##### `ReceiptBackup`
`receipts`テーブルに対応。最低限、マージに必要なメタも含む。
- `receiptId`（UUID）
- `receiptDatetime`（ISO 8601）
- `capturedAt`（ISO 8601 | null）
- `merchantName`（string）
- `totalAmountYen`（int）
- `paymentMethod`（string | null）
- `paymentServiceName`（string | null）
- `analysisStatus`（string）
- `needsReview`（boolean）
- `itemsSubtotalYen`（int）
- `adjustmentYen`（int）
- `createdAt` / `updatedAt`（ISO 8601）
- `deletedAt`（ISO 8601 | null）
- `deleteReason`（`MISTAKE|REFUND|OCR_MISS|DUPLICATE|OTHER` | null）
- `backupRevision`（int）

##### `ReceiptItemBackup`
`receipt_items`テーブルに対応。
- `itemId`（UUID）
- `receiptId`（UUID）
- `itemName`（string）
- `quantity`（int）
- `lineTotalYen`（int）
- `categoryMajor`（string）
- `categoryMinor`（string）
- `necessityScore`（int, 0〜100）
- `confidence`（number, 0.0〜1.0）
- `isAdjustment`（boolean）

##### マージ/復元ルール（バックアップ観点）
- 基本キーは `receiptId` / `itemId`。
- `receipts` のマージは `updatedAt` が新しい方を採用する（要件で確定）。
- `receiptItems` は `itemId` で突合し、親 `receiptId` が採用された側の集合として扱う（衝突時は親レシートの採用方針に従う）。
- 論理削除（`deletedAt`）もマージ対象（新しい `updatedAt` が勝つ）。

##### 将来オプション（バックアップに含める場合）
生データまで保持したい場合の追加フィールド。
- `geminiResults`: `GeminiResultBackup[]`（デバッグ/再現性目的。初期は含めない）

###### `GeminiResultBackup`
`gemini_results`テーブルに対応。
- `resultId`（UUID）
- `receiptId`（UUID）
- `schemaVersion`（string）
- `model`（string）
- `rawJson`（string）
- `createdAt`（ISO 8601）

### 14. 初回オンボーディング（2026-07-11 実装）

初回起動時（`onboarding_completed` 未設定）に **1回限り** のセットアップウィザードを表示する。完了後は通常の起動=カメラフローへ入る。

| ステップ | 内容 | スキップ |
|----------|------|----------|
| 1 ようこそ | アプリの一行説明 | 不可 |
| 2 APIキー | Gemini APIキー入力・保存（`GeminiApiKeyStore`） | 可（警告表示） |
| 3 カメラ | 撮影のためのカメラ権限（説明後にシステムダイアログ） | 拒否しても続行可 |
| 4 通知 | 解析完了通知のための権限（Android 13+ のみ） | 拒否しても続行可 |
| 5 完了 | 「はじめる」で通常 UI へ。バックアップは設定から JSON エクスポート可と1行案内 | 不可 |

**補完ガード（オンボーディング外）**

- APIキー未設定時: カメラタブにバナー表示 → 設定タブへ導線
- 送信確定時: APIキー未設定ならダイアログ（Phase 7.3）

### A. ナビゲーション（§3 入力/フロー）

| 項目 | 現要件 | 改定後 |

### B. 解析失敗時のリトライ（§10 Gemini連携、`analysis_queue`、Gemini厳格JSON）

**理由**: API 制限・タイムアウト等により解析失敗が1日数回発生する。自動リトライは API 消費と待ち時間を増やすだけで、ユーザーが失敗に気づきにくい。実利用者から **自動リトライ廃止・手動再送信** の要望があった。

**表示**: 失敗時は `analysisErrorMessage` を保持し、詳細・通知タブ等から再送信できること（Phase 6.2）。

**理由**: 修正画面で編集できる項目が少なく、レシート画像を見ながら品目を追加したいという実利用フィードバックがあった。§11 の「初期は編集範囲を絞る」は精度検証目的だったが、1か月運用で **手入力追加 UI への統一** が実用的と判断した。

## Gemini厳格JSONスキーマ（確定版・叩き台）
アプリは **このJSONのみを受け付ける**（厳格パース）。パース不能・必須欠落は **再試行→低信頼扱い**。

### 目的
- レシート画像から、**レシート（上位）** と **明細行（主）** を抽出し、各明細行にカテゴリ（大+小）と必須度スコア（0〜100）を付与する。
- **レシート合計**を正とし、明細合計との差分はアプリ側で「調整」行として補う（Geminiは差分行を出さない）。

### JSON仕様（型/必須）
#### ルート
- `schemaVersion`（必須）: `"1.0"`
- `receipt`（必須）: レシート上位情報
- `items`（必須）: 明細行配列（主出力、0件もあり得る）
- `warnings`（任意）: 解析上の注意（配列）

#### `receipt`
- `receiptDatetime`（必須）: レシート記載日時（ISO 8601文字列。例: `"2026-04-24T19:32:00+09:00"`）
- `capturedAt`（任意）: 撮影日時（ISO 8601文字列）
- `merchantName`（必須）: 店名（文字列）
- `totalAmountYen`（必須）: レシート合計（整数、円）
- `paymentMethod`（任意）: 支払手段（分類。後述のenum）
- `paymentServiceName`（任意）: 支払サービス名（自由テキスト。例: `"PayPay"`, `"Suica"`, `"楽天カード"`）

#### `items[]`
各要素（明細行）は以下を持つ。
- `lineIndex`（必須）: 明細の表示順（0始まりの整数。レシートの順序を保持）
- `itemName`（必須）: 商品名（文字列。取れない場合は `"不明"`）
- `quantity`（必須）: 数量（整数。取れない場合は `1`）
- `lineTotalYen`（必須）: 行合計（整数、円）
- `categoryMajor`（必須）: 大カテゴリ（固定リストのいずれか）
- `categoryMinor`（必須）: 小カテゴリ（大カテゴリに属する固定リストのいずれか）
- `necessityScore`（必須）: 必須度スコア（0〜100の整数）
- `confidence`（必須）: 明細行としての信頼度（0.0〜1.0の数値）

#### enum（固定）
##### `paymentMethod`
- `"CASH"`（現金）
- `"CREDIT_CARD"`（クレジットカード）
- `"DEBIT_CARD"`（デビットカード）
- `"TRANSPORT_IC"`（交通系IC）
- `"E_MONEY"`（電子マネー）
- `"QR_PAYMENT"`（QR決済）
- `"BANK_TRANSFER_OR_DEBIT"`（振込/銀行引落）
- `"UNKNOWN"`（不明）

##### `categoryMajor`
- `"FOOD"`（食費）
- `"DAILY_GOODS"`（日用品）
- `"TRANSPORT"`（交通）
- `"HOUSING"`（住居）
- `"UTILITIES"`（光熱）
- `"COMMUNICATION"`（通信）
- `"MEDICAL"`（医療）
- `"EDUCATION"`（教育）
- `"SOCIAL"`（交際）
- `"ENTERTAINMENT"`（娯楽）
- `"CLOTHING"`（衣服）
- `"OTHER"`（その他）

##### `categoryMinor`（大カテゴリ別の許容値・完全一致）
`categoryMinor` は以下の許容値の **いずれかに完全一致**すること（表記ゆれ不可）。
- `categoryMajor="FOOD"`: `スーパー` / `外食` / `その他`
- `categoryMajor="DAILY_GOODS"`: `衛生用品` / `雑貨（小物）` / `家具家電` / `その他`
- `categoryMajor="TRANSPORT"`: `電車/バス` / `定期券` / `その他`
- `categoryMajor="HOUSING"`: `家賃/住宅ローン` / `家具（大物）` / `その他`
- `categoryMajor="UTILITIES"`: `電気` / `ガス` / `水道` / `その他`
- `categoryMajor="COMMUNICATION"`: `携帯` / `インターネット` / `サブスク（通信系）` / `その他`
- `categoryMajor="MEDICAL"`: `病院` / `薬` / `健康/検診` / `その他`
- `categoryMajor="EDUCATION"`: `教材` / `その他`
- `categoryMajor="SOCIAL"`: `交際費` / `飲み会` / `贈り物` / `冠婚葬祭` / `その他`
- `categoryMajor="ENTERTAINMENT"`: `趣味` / `旅行` / `イベント` / `ゲーム（買い切り）` / `ゲーム（課金）` / `その他`
- `categoryMajor="CLOTHING"`: `衣類` / `靴/バッグ` / `クリーニング` / `その他`
- `categoryMajor="OTHER"`: `返済` / `その他`

## Roomエンティティ（確定版・叩き台）
目的は「撮影→解析→保存→一覧/集計→バックアップ」を **安定・拡張可能**に実装できる最小スキーマ。

### 設計方針
- **レシート上位（Receipt）** + **明細行（ReceiptItem）** の2段構造
- **論理削除**を基本とし、削除理由を保持
- **調整行**は明細行として保持するが、**UIには出さず集計専用**
- **Geminiの厳格JSON（生レスポンス）**も保存し、後で再現/デバッグ可能にする
- **解析キュー**（1件ずつ順番）をDBで管理し、アプリ再起動後も継続できる

### エンティティ一覧
#### 1) `receipts`（レシート上位）
1レシート=1レコード。一覧の基本単位。
- **主キー**
  - `receiptId`（TEXT/UUID）
- **時刻**
  - `receiptDatetime`（TEXT, ISO 8601, 必須）: レシート記載日時
  - `capturedAt`（TEXT, ISO 8601, 任意）: 撮影日時
  - `createdAt`（TEXT, ISO 8601, 必須）
  - `updatedAt`（TEXT, ISO 8601, 必須）
- **店舗/金額**
  - `merchantName`（TEXT, 必須）
  - `totalAmountYen`（INTEGER, 必須）
- **支払手段**
  - `paymentMethod`（TEXT, 任意）: enum（`CASH`/`CREDIT_CARD`/…/`UNKNOWN`）
  - `paymentServiceName`（TEXT, 任意）: 自由テキスト
- **解析状態（UI/運用）**
  - `analysisStatus`（TEXT, 必須）: enum（`PENDING`/`RUNNING`/`DONE`/`FAILED`/`NEEDS_REVIEW`）
  - `analysisStartedAt`（TEXT, 任意）
  - `analysisCompletedAt`（TEXT, 任意）
  - `analysisErrorMessage`（TEXT, 任意）
  - `needsReview`（INTEGER, 必須, 0/1）: 低信頼で確認が必要か
- **合計整合**
  - `itemsSubtotalYen`（INTEGER, 必須）: 明細（調整行を除く）行合計の合計
  - `adjustmentYen`（INTEGER, 必須）: レシート合計との差分（= 調整行の金額）
- **論理削除**
  - `deletedAt`（TEXT, 任意）
  - `deleteReason`（TEXT, 任意）: enum（`MISTAKE`/`REFUND`/`OCR_MISS`/`DUPLICATE`/`OTHER` など）
- **バックアップ/同期用**
  - `backupRevision`（INTEGER, 必須）: ローカル更新の世代（インクリメント）

**インデックス（推奨）**
- `receiptDatetime`（月次集計/一覧）
- `deletedAt`（除外検索）
- `analysisStatus`（キュー/一覧）

#### 2) `receipt_items`（明細行・主）
明細行（アイテム）を保持。必須度・カテゴリは **アイテム単位**。
- **主キー**
  - `itemId`（TEXT/UUID）
- **外部キー**
  - `receiptId`（TEXT, 必須）: `receipts.receiptId`
- **内容**
  - `lineIndex`（INTEGER, 必須）: 明細の表示順（0始まり）
  - `itemName`（TEXT, 必須）: 取れない場合は `"不明"`
  - `quantity`（INTEGER, 必須）: 取れない場合は `1`
  - `lineTotalYen`（INTEGER, 必須）
- **カテゴリ**
  - `categoryMajor`（TEXT, 必須）: enum（`FOOD`/`DAILY_GOODS`/…）
  - `categoryMinor`（TEXT, 必須）: 小カテゴリラベル（完全一致のみ許可）
- **必須度**
  - `necessityScore`（INTEGER, 必須）: 0〜100
- **信頼度**
  - `confidence`（REAL, 必須）: 0.0〜1.0
- **調整行フラグ**
  - `isAdjustment`（INTEGER, 必須, 0/1）: 1の場合はUIに出さず集計専用（カテゴリは`OTHER`/`その他`系）

**インデックス（推奨）**
- `receiptId`（結合・削除）
- `receiptId` + `lineIndex`（明細の表示順）
- `categoryMajor` + `categoryMinor`（集計）
- `necessityScore`（無駄遣いTOP抽出）

#### 3) `receipt_images`（端末内のレシート画像メタ）
バックアップ対象外。40日保持の削除ジョブに使う。
- **主キー**
  - `imageId`（TEXT/UUID）
- **外部キー**
  - `receiptId`（TEXT, 必須）
- **ファイル**
  - `localUri`（TEXT, 必須）: 端末内の保存先（URI/パス）
  - `mimeType`（TEXT, 必須）
  - `byteSize`（INTEGER, 必須）
  - `width`（INTEGER, 任意）
  - `height`（INTEGER, 任意）
- **保持期限**
  - `retentionUntil`（TEXT, 必須）: capturedAt基準で+40日など
- **状態**
  - `deletedAt`（TEXT, 任意）: 物理削除済みフラグ（ジョブの冪等性）

**インデックス（推奨）**
- `receiptId`
- `retentionUntil`

#### 4) `gemini_results`（Gemini生レスポンス）
厳格JSONの再現・調査・将来の再解析に使う。
- **主キー**
  - `resultId`（TEXT/UUID）
- **外部キー**
  - `receiptId`（TEXT, 必須）
- **内容**
  - `schemaVersion`（TEXT, 必須）: `"1.0"`
  - `model`（TEXT, 必須）: `"gemini-2.5-flash"` など
  - `rawJson`（TEXT, 必須）: 受信したJSON（厳格）
- **時刻**
  - `createdAt`（TEXT, 必須）

**インデックス（推奨）**
- `receiptId`

#### 5) `analysis_queue`（解析キュー）
1件ずつ順番に処理するためのキュー。WorkManagerと連携する想定。
- 送信前確認（フールセーフ）により、**ユーザーが送信確定したレシートのみ**をキューに追加する。
- **主キー**
  - `queueId`（TEXT/UUID）
- **対象**
  - `receiptId`（TEXT, 必須）
- **状態**
  - `status`（TEXT, 必須）: enum（`QUEUED`/`RUNNING`/`DONE`/`FAILED`）
  - `attemptCount`（INTEGER, 必須）
  - `lastError`（TEXT, 任意）
- **時刻**
  - `queuedAt`（TEXT, 必須）
  - `startedAt`（TEXT, 任意）
  - `finishedAt`（TEXT, 任意）

**インデックス（推奨）**
- `status` + `queuedAt`（次に処理する1件を高速取得）
- `receiptId`（重複防止）

### 集計ロジック上の注意（DB観点）
- レシートの集計は原則 `receipt_items` の `isAdjustment=0` を合計し、`adjustmentYen` を足して **`totalAmountYen` と一致**させる。
- 「調整行」は `receipt_items.isAdjustment=1` として保持し、一覧/明細UIには表示しない。
